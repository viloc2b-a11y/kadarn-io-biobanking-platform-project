# RC-12.3 Design Specification - Evidence Core Production Integration

**Date:** 2026-07-05  
**Status:** Proposed design - no implementation yet  
**Sprint:** RC-12.3 - Evidence Core Production Integration  
**Depends on:** RC-12.1 Persistent Portfolio, RC-12.2 Stability Runtime, RC-12.0 Sponsor Platform Freeze, RC-10.2 Sponsor Passport Contract, KUX-008, Sponsor Platform Architecture v1.0, Sponsor Platform Invariants v1.0

---

## Design Question

What parts of Sponsor Platform still do not consume Evidence Core through fully productive paths?

RC-12.3 is not a new feature sprint. It is a production integration sprint. Its purpose is to inventory, harden, or remove remaining production-ineligible integration paths between Sponsor Platform and Evidence Core while preserving public contract, UX, and architecture boundaries.

---

## 1. Background

RC-12.1 industrialized Sponsor Portfolio persistence.

RC-12.2 industrialized Stability Runtime.

At the end of RC-12.2, Sponsor Passport has productive internal runtime coverage for:

- persistent portfolio membership and access scope;
- Evidence Core-backed claims;
- Evidence Core-backed capability grouping;
- Evidence Core-backed provenance detail;
- audit-backed history;
- conservative recommendation derivation;
- Evidence Core-backed stability derivation.

The remaining RC-12.3 risk is not that every section lacks Evidence Core integration. Most sections already have it. The risk is that production deployment can still select mock runtime paths by default, stale route comments still describe old mock behavior, and some adjacent sponsor surfaces remain provisional or outside the Sponsor Passport production path.

RC-12.3 must therefore begin with an integration inventory. If the inventory shows a component is already productive, RC-12.3 must not modify it only "por si acaso".

### Inventory Verdict

Most Sponsor Passport sections already consume productive Evidence Core or approved persistent repository sources.

The primary RC-12.3 implementation candidates are:

- production source selection hardening;
- `.env.example` / deployment guidance for `SPONSOR_PASSPORT_DATA_SOURCE`;
- stale route comment cleanup;
- production-mode test coverage;
- deterministic missing-data behavior documentation.

The primary non-candidates are:

- rewriting claims, provenance, history, recommendations, stability, or portfolio "just in case";
- removing mock mode entirely;
- changing Sponsor UI or contract;
- absorbing Sponsor Search into Passport scope without a separate program decision.

## 3. Remaining Temporary Integrations

### 3.1 Runtime Source Selection Defaults to Mock

| Field | Value |
|---|---|
| Location | `apps/api/src/lib/sponsor-passport/factory.ts` |
| Current behavior | `SPONSOR_PASSPORT_DATA_SOURCE` supports `mock` or `evidence-core`; missing env defaults to `mock`. |
| Why temporary | A production deployment could accidentally serve mock Sponsor Passport data if the env is omitted. |
| Replacement | Make production behavior explicit: either require `SPONSOR_PASSPORT_DATA_SOURCE=evidence-core` in production, flip production default, or reject `mock` when `NODE_ENV=production` unless explicitly allowed. |
| Impact | Production hardening only; no contract shape change. |
| Rollback | Preserve explicit `mock` mode for local/demo/test environments. |

### 3.2 Missing Environment Documentation

| Field | Value |
|---|---|
| Location | `apps/api/.env.example` |
| Current behavior | Does not document `SPONSOR_PASSPORT_DATA_SOURCE`. |
| Why temporary | Runtime source policy is not discoverable to deployers. |
| Replacement | Add documented local and production guidance. |
| Impact | Configuration documentation only. |
| Rollback | Revert env example/comment if production source selection policy changes. |

### 3.3 Stale Sponsor Passport Route Comments

| Field | Value |
|---|---|
| Location | Sponsor Passport API route headers under `apps/api/src/app/api/v1/sponsor/passports*`. |
| Current behavior | Comments still describe RC-10.3 mock/no-DB behavior even though routes use `PassportStore`. |
| Why temporary | Documentation drift can mislead maintainers into thinking routes are still mock-only. |
| Replacement | Update comments to describe source-agnostic `PassportStore` routing. |
| Impact | No runtime behavior change. |
| Rollback | Not needed; comments should reflect current architecture. |

### 3.4 Production-Mode Test Gap

| Field | Value |
|---|---|
| Location | `tests/api/sponsor-passport-store.test.ts`, Sponsor Passport route/store tests. |
| Current behavior | Store-level tests cover env switching and Evidence Core store behavior; HTTP route-level evidence-core mode coverage is limited. |
| Why temporary | Production source selection hardening needs direct test coverage so regressions do not reintroduce mock defaults. |
| Replacement | Add tests that assert production mode selects or requires Evidence Core as designed. |
| Impact | Test coverage only; no contract shape change. |
| Rollback | Tests can be adjusted if the production source selection policy changes. |

### 3.5 Sponsor Search Provisional Capability Placeholder

| Field | Value |
|---|---|
| Location | `apps/api/src/app/api/v1/sponsor/search/route.ts` |
| Current behavior | Registers institutions with `capabilities: []` and notes production should use cached engine outputs. |
| Why temporary | It is a provisional adjacent Sponsor Search surface, not the RC-10.2 Sponsor Passport runtime. |
| Replacement | Separate design decision if Sponsor Search becomes part of Evidence Core production integration. |
| Impact | Out of RC-12.3 Sponsor Passport scope by default. |
| Rollback | Keep unchanged unless RC-12.3 scope is explicitly expanded. |

## 4. Target Architecture

### Production Sponsor Passport Flow

```
Sponsor request
   -> Sponsor API route
   -> auth + sponsor org membership
   -> PassportStore
   -> EvidenceCorePassportStore
   -> SponsorPortfolioRepository
   -> Evidence Core reads
   -> Adapter mappers
   -> RC-10.2 Sponsor Passport DTO
```

### Source Ownership

| Source | Owns |
|---|---|
| Sponsor Portfolio Repository | Sponsor-owned portfolio membership and access scope. |
| Evidence Core | Claims, evidence, graph/provenance, lifecycle audit facts, temporal evidence state. |
| Sponsor Passport Adapter | Contract mapping only. |
| Sponsor API | Auth, authorization, rate limit, route params, envelope. |
| Sponsor UI | Rendering only; no Evidence Core or adapter dependency. |

### Allowed Non-Production Flow

```
Local/demo/test request
   -> Sponsor API route or unit test
   -> PassportStore
   -> MockPassportStore
   -> RC-10.2-compatible mock fixtures
```

Mock mode may remain, but it must be explicit and isolated from production.

## 6. Adapter Impact

### Methods That Must Remain Stable

`PassportStore` remains unchanged:

```ts
getPortfolioIndex(sponsorOrgId: string): Promise<PassportPortfolioIndexResponse>
getInstitutionalPassport(sponsorOrgId: string, institutionId: string): Promise<InstitutionalPassport | undefined>
getClaimProvenanceDetail(sponsorOrgId: string, institutionId: string, claimId: string): Promise<PassportClaimProvenanceDetail | undefined>
isInstitutionInPortfolio(sponsorOrgId: string, institutionId: string): Promise<boolean>
```

### Expected Adapter Changes

RC-12.3 should avoid adapter rewrites unless the inventory identifies real production debt.

Allowed adapter changes:

- add production-mode tests around existing methods;
- remove stale comments that still describe mock-only behavior;
- add explicit source-selection guardrails around factory behavior.

Not allowed without separate approval:

- changing DTO fields;
- changing `PassportStore` signatures;
- moving UI concerns into adapter;
- bypassing `PassportStore`;
- reusing Public Projection or `continuity/passport`;
- replacing productive Evidence Core mappings without a demonstrated defect.

---

## 7. Runtime Impact

### What Changes

- Production source selection becomes explicit and safer.
- Deployers get documented environment guidance.
- Tests assert production mode does not silently choose mock data.
- Stale mock-era comments are removed from source-agnostic route code.

### What Does Not Change

- Sponsor Passport public response shape.
- Allowed enum values.
- Sponsor UI behavior.
- Sponsor API route paths.
- Portfolio repository behavior.
- Claims, capabilities, provenance, history, recommendations, and stability mapping semantics.
- Mock mode availability for local/demo/test when explicitly selected.

## 8. Failure Model

| Failure | Expected Behavior |
|---|---|
| Evidence Core source unavailable | API should use existing error handling/envelope; do not silently fall back to mock in production. |
| Evidence Core returns partial claim/evidence data | Adapter maps available facts conservatively; missing detail should not be invented. |
| Institution not in sponsor portfolio | Existing authorization behavior remains: no passport detail is returned. |
| Institution in portfolio but no claims | Existing behavior remains deterministic: omitted from index and no passport detail. |
| Audit events unavailable | Runtime sections depending on audit should degrade according to existing mapper behavior; no fabricated history/review facts. |
| Source selection env missing in production | Target behavior should fail closed or select Evidence Core explicitly, based on approved implementation policy. |
| Mock mode requested in production | Target behavior should reject or require explicit override, based on approved implementation policy. |
| Inconsistent Evidence Core data | Adapter should preserve provenance and avoid scoring/inference beyond approved mappers. |

## 10. Exit Criteria

RC-12.3 is complete only when:

- all production Sponsor Passport reads resolve to Evidence Core-backed paths or approved persistent repositories;
- mock mode is explicit, isolated, and production-safe;
- no production path silently falls back to mock data;
- all remaining temporary integrations are removed, guarded, or explicitly documented as out of scope;
- Sponsor Passport keeps the exact same RC-10.2 public contract;
- KUX-008 and Sponsor UI behavior remain unchanged;
- `PassportStore` remains the API-to-runtime boundary;
- Evidence Core remains the source of truth for claims, evidence, provenance, lifecycle audit facts, and related runtime data;
- Sponsor Search provisional behavior is either explicitly out of scope or handled through a separate approved design;
- QA gates pass and inherited findings are documented.

## 12. Scope Control Rule

RC-12.3 is inventory-driven.

If a component is already productive, contract-compatible, and behind the approved boundaries, it should not be changed only because it was inspected. RC-12.3 should eliminate real production integration debt, not create review churn.

The expected center of gravity is production source selection and mock isolation, not remapping every Sponsor Passport section.
