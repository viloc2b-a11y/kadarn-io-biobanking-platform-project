# RC-12.1 Design Specification - Persistent Portfolio

**Date:** 2026-07-05  
**Status:** Proposed design - no implementation yet  
**Sprint:** RC-12.1 - Persistent Portfolio  
**Depends on:** RC-12.0 Sponsor Platform Freeze, RC-10.2 Sponsor Passport Contract, Sponsor Platform Architecture v1.0, Sponsor Platform Invariants v1.0

---

## Design Question

How does Sponsor Portfolio move from a temporary representation to a persistent production asset without altering any public contract?

RC-12.1 replaces temporary allowlists and embedded JSON with persistent portfolio infrastructure. The source of portfolio data changes; the Sponsor Passport API contract and user experience do not.

---

## 1. Background

RC-11 implemented Sponsor Passport as a functional Evidence Core-backed runtime. Portfolio scoping was intentionally implemented as a temporary allowlist so the adapter could close without introducing new persistence.

Current temporary components:

| Component | Current Behavior | Temporary Aspect |
|---|---|---|
| `portfolio-allowlist.ts` | Parses `SPONSOR_PASSPORT_PORTFOLIO_JSON` and test overrides. | Environment JSON, no durable ownership model. |
| `map-portfolio-index.ts` | Builds portfolio index from allowlist entries plus Evidence Core reads. | Portfolio membership source is not persistent. |
| `isInstitutionInPortfolio()` | Checks allowlist membership. | No database-backed authorization asset. |
| Allowlist metadata | `displayName`, `location`, `memberSince`. | Embedded values may diverge from canonical organization data. |

RC-12.1 exists to make portfolio membership a durable production concept while preserving the `PassportStore` interface and RC-10.2 response shape.

---

## 2. Objectives

| Objective | Requirement |
|---|---|
| Remove allowlists | Production mode must not depend on `SPONSOR_PASSPORT_PORTFOLIO_JSON`. |
| Remove embedded JSON | Portfolio membership must come from persistent storage. |
| Persist portfolio membership | Sponsor-to-institution membership becomes a durable platform asset. |
| Preserve RC-10.2 | Response DTOs and API semantics remain unchanged. |
| Preserve KUX-008 | Institutional Passport user experience remains unchanged. |
| Preserve Sponsor Platform v1.0 | Dependency rules and adapter boundary remain intact. |
| Preserve Evidence Core truth | Claims/evidence/provenance remain owned by Evidence Core. |
| Preserve mock mode policy | Mock/local behavior remains explicit and isolated. |

---

## 3. Non-Goals

RC-12.1 does not:

- modify Sponsor Passport contract fields;
- modify Sponsor UI;
- introduce new public Sponsor Passport endpoints;
- change Evidence Core claim/evidence rules;
- change authorization semantics beyond replacing membership storage;
- implement Stability Runtime;
- change recommendations;
- touch Public Projection;
- touch `continuity/passport`;
- introduce Sponsor Intelligence features;
- redesign organization membership.

## 4. Current Architecture

### Current Portfolio Read Flow

```
Sponsor request
   -> Sponsor API
   -> require active sponsor org membership
   -> PassportStore.getPortfolioIndex(sponsorOrgId)
   -> EvidenceCorePassportStore.getPortfolioIndex()
   -> buildPortfolioIndex()
   -> getAllowedInstitutions(sponsorOrgId)
   -> SPONSOR_PASSPORT_PORTFOLIO_JSON / test override
   -> readInstitutionEvidence(institutionId)
   -> PassportPortfolioIndexResponse
```

### Current Detail Authorization Flow

```
Sponsor request
   -> Sponsor API
   -> sponsorOrgId from active org
   -> institutionId from route params
   -> PassportStore.getInstitutionalPassport(sponsorOrgId, institutionId)
   -> checkInstitutionInPortfolio()
   -> isInstitutionInPortfolioAllowlist()
   -> Evidence Core reads if allowed
```

### Current Temporary Data Location

| Data | Location |
|---|---|
| Sponsor portfolio membership | `SPONSOR_PASSPORT_PORTFOLIO_JSON` |
| Test portfolio membership | `setPortfolioAllowlistForTests()` |
| Institution display fallback | Allowlist entry or institution ID |
| Institution facts | Evidence Core reads + `organizations` table |

## 5. Target Architecture

### Target Read Flow

```
Sponsor request
   -> Sponsor API
   -> require active sponsor org membership
   -> PassportStore.getPortfolioIndex(sponsorOrgId)
   -> EvidenceCorePassportStore.getPortfolioIndex()
   -> Portfolio repository
   -> persistent portfolio memberships
   -> Evidence Core institution reads
   -> PassportPortfolioIndexResponse
```

### Target Authorization Flow

```
Sponsor request
   -> Sponsor API
   -> sponsorOrgId from active org
   -> institutionId from route params
   -> PassportStore.getInstitutionalPassport(sponsorOrgId, institutionId)
   -> persistent portfolio membership check
   -> Evidence Core reads if allowed
```

### Current vs Target

| Concern | Current | Target |
|---|---|---|
| Membership source | Environment JSON/test override | Persistent tables |
| Ownership | Implicit sponsor org key in JSON | Explicit sponsor organization ownership |
| Versioning | None | Version/audit fields on membership changes |
| Authorization check | Allowlist lookup | Repository lookup by sponsor org + institution org |
| Contract | RC-10.2 | Unchanged |
| UI | Sponsor UI consumes API | Unchanged |
| Evidence Core claims/evidence | Evidence Core | Unchanged |

## 6. Data Model

Portfolio is a Sponsor Platform access asset that references Evidence Core-compatible organization identifiers. It does not own claim or evidence facts.

### Canonical Entities

| Entity | Purpose |
|---|---|
| `sponsor_portfolios` | Root portfolio owned by a sponsor organization. |
| `sponsor_portfolio_memberships` | Institution/site membership within a sponsor portfolio. |
| `organizations` | Existing canonical organization/site identity table. |
| `audit_events` | Existing audit trail for lifecycle events where applicable. |

### Proposed Tables

#### `sponsor_portfolios`

| Column | Type | Notes |
|---|---|---|
| `id` | UUID PK | Stable portfolio ID. |
| `sponsor_org_id` | UUID FK -> `organizations(id)` | Owner sponsor organization. |
| `name` | TEXT | Internal name; not exposed in RC-10.2 response. |
| `status` | TEXT or enum | `active`, `archived`. |
| `version` | INTEGER | Increment on material membership changes. |
| `created_at` | TIMESTAMPTZ | Audit support. |
| `updated_at` | TIMESTAMPTZ | Audit support. |
| `created_by` | UUID nullable | User who created the portfolio where available. |

#### `sponsor_portfolio_memberships`

| Column | Type | Notes |
|---|---|---|
| `id` | UUID PK | Stable membership ID. |
| `portfolio_id` | UUID FK -> `sponsor_portfolios(id)` | Parent portfolio. |
| `sponsor_org_id` | UUID FK -> `organizations(id)` | Denormalized owner for fast policy checks. |
| `institution_org_id` | UUID FK -> `organizations(id)` | Site/institution org ID used by Evidence Core reads. |
| `status` | TEXT or enum | `active`, `removed`, `paused`. |
| `member_since` | DATE | Source for `PassportInstitutionSummary.memberSince`. |
| `display_name_override` | TEXT nullable | Optional; canonical org name preferred. |
| `location_override` | TEXT nullable | Optional; canonical org location preferred. |
| `version` | INTEGER | Increment on membership metadata changes. |
| `created_at` | TIMESTAMPTZ | Audit support. |
| `updated_at` | TIMESTAMPTZ | Audit support. |
| `removed_at` | TIMESTAMPTZ nullable | Soft removal for history/audit. |

### Relationships

```
organizations (sponsor)
        |
        | 1:N
        v
sponsor_portfolios
        |
        | 1:N
        v
sponsor_portfolio_memberships
        |
        | N:1
        v
organizations (institution/site)
        |
        v
Evidence Core claims/evidence by organization_id
```

### Keys and Constraints

| Constraint | Purpose |
|---|---|
| Unique active portfolio per sponsor org, if business policy requires it | Prevent ambiguous default reads. |
| Unique active membership by `(portfolio_id, institution_org_id)` | Prevent duplicate institutions in one portfolio. |
| `sponsor_org_id` must match parent portfolio owner | Preserve ownership integrity. |
| `institution_org_id` references `organizations(id)` | Preserve Evidence Core-compatible institution identity. |
| Soft delete through status/removed_at | Preserve auditability and rollback. |

### Indexes

| Index | Purpose |
|---|---|
| `(sponsor_org_id, status)` on `sponsor_portfolios` | Resolve active sponsor portfolio. |
| `(sponsor_org_id, institution_org_id, status)` on memberships | Fast authorization check. |
| `(portfolio_id, status)` on memberships | Portfolio index reads. |
| `(institution_org_id)` on memberships | Reverse lookup/audit support. |

## 7. Migration Strategy

### Source

Current source is `SPONSOR_PASSPORT_PORTFOLIO_JSON`.

### Proposed Migration Path

1. Add persistent tables.
2. Add a one-time seed/import script or migration helper for known allowlist entries.
3. Validate imported membership counts per sponsor organization.
4. Switch production portfolio reads to persistent repository.
5. Keep test helpers explicit and separate from production runtime.
6. Remove `SPONSOR_PASSPORT_PORTFOLIO_JSON` from production path.

### Validation

| Validation | Expected Result |
|---|---|
| Sponsor org count | Matches imported source set. |
| Membership count | Matches allowlist entries. |
| Institution IDs | Exist in `organizations`. |
| Portfolio index response | Contract-compatible with pre-migration response. |
| Detail authorization | Allows same institutions as imported membership set. |
| Negative authorization | Denies institutions not in persistent membership. |

### Rollback

Preferred rollback is data-level:

- preserve old allowlist source until persistent path is verified in non-production;
- keep imported records soft-deletable;
- if production rollout fails, revert application read path to the previous commit rather than silently falling back at runtime.

### Double Read vs Direct Cutover

RC-12.1 should use a direct cutover after validation, not long-lived double read.

Short-lived comparison mode may be used in tests or migration verification, but production runtime should have a single authoritative portfolio source after RC-12.1 closure.

## 8. Adapter Impact

### Interfaces That Remain Stable

`PassportStore` remains unchanged:

```ts
getPortfolioIndex(sponsorOrgId: string): Promise<PassportPortfolioIndexResponse>
getInstitutionalPassport(sponsorOrgId: string, institutionId: string): Promise<InstitutionalPassport | undefined>
getClaimProvenanceDetail(sponsorOrgId: string, institutionId: string, claimId: string): Promise<PassportClaimProvenanceDetail | undefined>
isInstitutionInPortfolio(sponsorOrgId: string, institutionId: string): Promise<boolean>
```

### Methods That Change Internally

| Method | Current Source | Target Source |
|---|---|---|
| `getPortfolioIndex()` | Allowlist entries | Persistent repository memberships |
| `isInstitutionInPortfolio()` | Allowlist lookup | Persistent membership query |
| `resolvePortfolioEntry()` | Allowlist entry metadata | Persistent membership + organization metadata |

### Temporary Code To Remove or Isolate

| Code | RC-12.1 Decision |
|---|---|
| `SPONSOR_PASSPORT_PORTFOLIO_JSON` production parsing | Remove from production path. |
| `getAllowedInstitutions()` allowlist runtime | Replace with repository read. |
| `isInstitutionInPortfolioAllowlist()` | Replace with repository membership check. |
| `setPortfolioAllowlistForTests()` | Replace with DB-backed test fixtures or keep only if explicitly scoped to mock/unit tests. |

### New Adapter Concepts

| Concept | Purpose |
|---|---|
| Portfolio repository/read query | Resolve sponsor portfolio and memberships. |
| Membership record mapper | Map persistent row + organization data to existing portfolio summary inputs. |
| Portfolio test fixture helper | Seed persistent membership data in tests without env JSON. |

## 9. Runtime Flow

### Read Flow

```
Sponsor
   -> Sponsor API
   -> requireOrgMembership()
   -> PassportStore
   -> Portfolio repository
   -> persistent portfolio memberships
   -> Evidence Core organization/claim/evidence reads
   -> Adapter mappers
   -> RC-10.2 contract response
   -> Sponsor UI
```

### Write Flow

RC-12.1 does not require a new public write API.

If portfolio updates are needed in this sprint, they should be handled as internal seed/admin operations, not new Sponsor Passport public endpoints.

Any future public write flow requires separate design and authorization review.

## 10. Failure Model

| Failure | Expected Behavior |
|---|---|
| No portfolio for sponsor org | Portfolio index returns empty `items`; detail reads return not found/unauthorized equivalent through existing route behavior. |
| Portfolio exists but has no active memberships | Portfolio index returns empty `items`. |
| Institution membership missing | Detail/provenance read returns no passport/provenance through existing route behavior. |
| Institution organization missing | Membership is ignored for index; detail read returns not found. |
| Evidence Core has no claims for institution | Existing behavior preserved: no passport detail. |
| Repository timeout/error | API returns existing error envelope; no partial data invention. |
| Duplicate active membership | Prevented by constraint; if encountered, fail closed and alert/test failure. |
| Inconsistent sponsor ownership | Fail closed; membership must not authorize access. |

No failure mode may bypass authorization or expose evidence outside sponsor scope.

## 12. Exit Criteria

RC-12.1 is complete only when:

- portfolio data no longer comes from production allowlists or embedded JSON;
- portfolio membership is persistent;
- ownership is enforced by sponsor organization;
- `PassportStore` interface remains unchanged;
- RC-10.2 response shape remains unchanged;
- Sponsor UI behavior is indistinguishable to consumers;
- portfolio index and detail authorization are backed by persistent membership;
- mock/test behavior is explicit and not confused with production runtime;
- all technical gates pass;
- a completion report records decisions and residual risks.

## 13. Architectural Impact Assessment

| Question | Expected Answer |
|---|---|
| Does this break any Sponsor Platform invariant? | No. It should strengthen persistence while preserving boundaries. |
| Does this modify a layer dependency? | No. Reads remain API -> `PassportStore` -> adapter/repository. |
| Does this introduce technical debt? | Minimal if allowlist code is removed or explicitly isolated. |
| Does this require an ADR? | Not if it follows this specification; yes if ownership, contract, or dependency rules change. |
| Does this affect the public contract? | No. RC-10.2 remains unchanged. |
| Can this be reused by future modules? | Yes. Persistent sponsor-owned membership can support future Sponsor Intelligence if kept behind approved APIs. |
| Does this change Evidence Core domain policy? | No. Portfolio is access scope; claims/evidence remain Evidence Core-owned. |
