**Date:** 2026-07-05

## Executive Summary

RC-12.2 replaced the provisional Sponsor Passport stability runtime with a deterministic, evidence-backed lifecycle runtime while preserving the RC-10.2 Sponsor Passport contract and Sponsor UI behavior.

Before RC-12.2, Evidence Core mode returned `Under Review` as a universal placeholder in portfolio index and passport detail. After RC-12.2, Stability is derived from productive claim, evidence, audit, temporal, and review signals through an internal domain model and Source Layer. The public output remains the existing RC-10.2 `StabilityIndicator` enum.

**Recommended verdict:** RC-12.2 is ready to close after acceptance of this report.

## 2. Architecture Before and After

### Before RC-12.2

```
Sponsor request
   -> Sponsor API
   -> PassportStore
   -> EvidenceCorePassportStore
   -> Evidence Core reads
   -> DEFAULT_STABILITY / STUB_STABILITY
   -> stability: 'Under Review'
   -> RC-10.2 Sponsor Passport DTOs
```

### After RC-12.2

```
Sponsor request
   -> Sponsor API
   -> PassportStore
   -> EvidenceCorePassportStore
   -> persistent portfolio membership check
   -> Evidence Core reads
   -> Stability Source Layer
   -> Stability Domain Evaluator
   -> RC-10.2 StabilityIndicator
   -> RC-10.2 Sponsor Passport DTOs
```

### Boundary Changes

| Concern | Before | After |
|---|---|---|
| Portfolio index stability | Universal `Under Review` stub | Derived from institution claim/evidence/audit signals |
| Passport detail stability | Universal `Under Review` stub | Derived from full institution read and audit trail |
| Stability rules | Absent from production runtime | Internal domain model with deterministic transition priority |
| Source data | Not consumed for Stability | Claims, evidence nodes, audit events, temporal decay, review signals |
| Contract | Existing enum | Unchanged |
| UI | Existing Sponsor UI | Unchanged |
| Public API routes | Existing Sponsor routes | Unchanged |
| Portfolio repository | RC-12.1 persistent access scope | Unchanged |
| Evidence Core ownership | Claims/evidence/audit owner | Unchanged |

## 5. Architectural Decisions

| Decision | Rationale |
|---|---|
| Keep the public enum unchanged | RC-10.2 already defines `Stable`, `Evolving`, `Under Review`, and `Evidence Refresh Needed`; RC-12.2 changes derivation, not contract shape. |
| Model Stability as knowledge state | KUX-007/KUX-008 define Stability as the state of Kadarn's knowledge, never institution quality. |
| Use deterministic priority rules | Review-sensitive state must win over stale/fresh signals; behavior must be testable and explainable. |
| Keep Stability internal to Sponsor Passport runtime | No public explanation field was added; any future exposure requires contract governance. |
| Reuse Evidence Core facts, not new truth | Stability reads claims, evidence, audit, temporal decay, and review signals; it does not mutate Evidence Core or invent facts. |
| Keep portfolio as access scope only | RC-12.1 repository membership authorizes reads but does not influence the stability value. |
| Preserve `Under Review` as a legitimate state | `Under Review` is no longer universal, but remains valid when review-source signals exist. |
| Avoid scoring | Stability is categorical and non-ordinal; no numeric rollups, institution rankings, badges, certifications, or AI were introduced. |
| Separate Source Layer from Domain Evaluator | Source mapping can evolve independently from transition rules while preserving deterministic domain behavior. |
