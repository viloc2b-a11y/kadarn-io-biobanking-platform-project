# RC-12.2 Design Specification - Stability Runtime

**Date:** 2026-07-05  
**Status:** Proposed design - no implementation yet  
**Sprint:** RC-12.2 - Stability Runtime  
**Depends on:** RC-12.1 Persistent Portfolio, RC-12.0 Sponsor Platform Freeze, RC-10.2 Sponsor Passport Contract, KUX-007, KUX-008, Sponsor Platform Architecture v1.0, Sponsor Platform Invariants v1.0

---

## Design Question

How does Sponsor Passport replace the provisional `Under Review` stability placeholder with a productive lifecycle runtime without changing the public contract or user experience?

RC-12.2 replaces the universal Evidence Core-mode stability stub with deterministic knowledge-state rules. The output remains the existing RC-10.2 `StabilityIndicator` enum.

---

## 1. Objective

RC-12.2 exists to make `stability` a real runtime signal.

| Objective | Requirement |
|---|---|
| Replace universal stub behavior | Evidence Core mode must not always return `Under Review`. |
| Preserve RC-10.2 | No new fields, removed fields, renamed fields, or enum changes. |
| Preserve KUX-008 | Stability remains a knowledge-state indicator, not a quality grade. |
| Preserve Sponsor Platform architecture | Stability stays behind `PassportStore` and adapter boundaries. |
| Preserve RC-12.1 Repository Layer | Portfolio persistence and membership logic are not redesigned. |
| Use productive inputs | Stability is derived from existing claim, evidence, audit, lifecycle, and temporal state. |
| Keep logic deterministic | No AI, no numeric scoring, no institution ranking. |

---

## 2. Non-Goals

RC-12.2 does not:

- modify the Sponsor Passport API contract;
- add public stability fields;
- add numeric scores, percentages, weights, grades, badges, or rankings;
- certify, verify, or endorse institutions;
- change KUX-008 copy or Sponsor UI layout;
- redesign Portfolio persistence from RC-12.1;
- alter Evidence Core claim/evidence semantics;
- introduce AI-generated stability;
- touch Public Projection;
- touch `continuity/passport`;
- implement new sponsor recommendations.

---

## 3. What Stability Means

Stability is the state of Kadarn's knowledge about an institution at a point in time.

It is not:

- an institution quality label;
- a performance grade;
- a certification state;
- a trust score;
- a completeness meter;
- a recommendation.

### Allowed Contract Values

The public output remains the existing RC-10.2 enum:

| Value | Meaning |
|---|---|
| `Stable` | Evidence is current, there is no material recent movement, and no claim/evidence state is contested or pending review. |
| `Evolving` | Evidence, claims, or lifecycle state are actively changing. New or updated knowledge exists and is still settling. |
| `Under Review` | Human judgment or review is in progress, especially for contested state, open curation, counter-evidence, or response workflows. |
| `Evidence Refresh Needed` | Evidence is missing, insufficient, aging, decayed, or beyond attention thresholds. |

### Product Law

Stability values are categorical and non-ordinal. They must never be used to rank institutions or imply that one institution is better than another.

---

## 4. Current Behavior

### Portfolio Index

`apps/api/src/lib/sponsor-passport/adapter/map-portfolio-index.ts` currently uses:

```ts
const DEFAULT_STABILITY: StabilityIndicator = 'Under Review'
```

Every Evidence Core-backed portfolio summary receives the same provisional value.

### Passport Detail

`apps/api/src/lib/sponsor-passport/evidence-core-passport-store.ts` currently uses:

```ts
const STUB_STABILITY: StabilityIndicator = 'Under Review'
```

Every Evidence Core-backed passport detail receives the same provisional value.

### Mock Mode

Mock fixtures already contain mixed stability values. Mock mode may remain contract-preserving, but production Evidence Core mode must use the Stability Runtime.

---

## 5. Target Architecture

### Target Runtime Flow

```
Sponsor request
   -> Sponsor API
   -> PassportStore
   -> EvidenceCorePassportStore
   -> persistent portfolio membership check
   -> Evidence Core reads
   -> Stability Runtime
   -> adapter mapping
   -> RC-10.2 StabilityIndicator
```

### Component Boundary

| Component | Responsibility |
|---|---|
| `PassportStore` | Keeps route-level callers independent of stability implementation. |
| `EvidenceCorePassportStore` | Orchestrates reads and invokes stability runtime. |
| Stability Runtime | Determines one enum value from allowed inputs. |
| Adapter mappers | Format existing passport DTO sections. |
| Evidence Core | Owns claims, evidence, audit, process, and lifecycle facts. |
| Portfolio Repository | Authorizes sponsor access only; not a stability engine. |

### Dependency Rule

Stability Runtime may depend on Evidence Core read models and audit/lifecycle records through adapter-approved query functions. It must not depend on Sponsor UI, public projection, continuity passport, or RC-12.1 repository internals beyond the already enforced access gate.

---

## 6. Source Data

Stability should be derived from existing, explainable data:

| Source | Example Inputs | Use |
|---|---|---|
| Claims | Claim status, claim confidence, updated timestamps, contested state | Detect movement, insufficiency, and review state. |
| Evidence nodes | Evidence class, date, weight, positive/negative support | Detect freshness, decay, counter-evidence, and gaps. |
| Capability temporal state | `fresh`, `aging`, `decayed` from existing capability logic | Detect refresh pressure. |
| Audit events | Claim/evidence lifecycle events | Detect recent movement and review activity. |
| Response/counter-evidence workflows | Counter-evidence and right-of-response state where available | Detect human review state. |
| Portfolio membership | Access boundary only | Must not influence the stability value except by allowing the read. |

### Data That Must Not Be Used

| Data | Reason |
|---|---|
| Institution quality assumptions | Violates KUX knowledge-state law. |
| Numeric aggregate scores | Violates RC-10.2 and Sponsor Platform invariants. |
| Sponsor preference or ranking | Stability is not feasibility scoring. |
| Public Projection fields | Wrong audience and contract boundary. |
| `continuity/passport` state | Wrong product surface. |

---

## 7. Lifecycle Model

The lifecycle describes the state of knowledge, not the institution.

```
Sparse / missing evidence
   -> Evidence Refresh Needed

Fresh evidence with no movement or contestation
   -> Stable

Recent evidence, claim, or audit movement
   -> Evolving

Counter-evidence, contested state, open response, or curation
   -> Under Review

Aging or decayed evidence
   -> Evidence Refresh Needed
```

### Lifecycle Inputs

| Lifecycle Signal | Example |
|---|---|
| Missing support | Claim has no supporting evidence. |
| Insufficient confidence | Claim maps to `Insufficient`. |
| Aging evidence | Supporting evidence approaches decay horizon. |
| Decayed evidence | Supporting evidence is past decay horizon. |
| Recent movement | Claim or evidence changed within a configured recent window. |
| Contestation | Negative evidence, counter-evidence, contested claim, or response flow. |
| Quiet current state | Evidence is current and no review/movement signal exists. |

---

## 8. Transition Rules

Rules must be deterministic and priority-based.

### Proposed Priority

| Priority | Result | Condition |
|---:|---|---|
| 1 | `Under Review` | Any claim/evidence state is contested, counter-evidence exists, or human review/right-of-response/curation is open. |
| 2 | `Evidence Refresh Needed` | Evidence is missing, confidence is `Insufficient`, or any material capability is `decayed`. |
| 3 | `Evolving` | Recent claim/evidence/audit movement exists or any material capability is `aging`. |
| 4 | `Stable` | Claims are supported, evidence is fresh, no contestation exists, and no recent movement exists. |

### Why Priority Matters

If multiple signals exist, the most review-sensitive state wins:

- contested and decayed -> `Under Review`;
- insufficient and recent movement -> `Evidence Refresh Needed`;
- aging and fresh elsewhere -> `Evolving`;
- no signals -> `Stable`.

This avoids hidden risk and preserves the KUX principle that challenged or stale knowledge must be visible.

### Configuration

The first implementation should use conservative constants in code, not tenant-specific configuration:

| Constant | Proposed Use |
|---|---|
| Recent movement window | Determines `Evolving` from recent audit/claim/evidence updates. |
| Aging threshold | Reuse existing capability temporal logic where possible. |
| Material claim rule | Start with any visible claim/capability, not weighted portfolio scoring. |

Any later configuration system requires a separate governance decision.

---

## 9. Failure Model

| Failure | Expected Behavior |
|---|---|
| No claims for institution | Existing behavior remains: no passport detail where current runtime returns none. |
| Claims exist but no supporting evidence | `Evidence Refresh Needed`. |
| Evidence dates are missing or unparsable | Treat as `Evidence Refresh Needed` or `Evolving` only if lifecycle/audit movement supports it; never silently mark `Stable`. |
| Audit events unavailable | Derive from claims/evidence only; do not fail open to `Stable`. |
| Stability runtime throws unexpectedly | API should use existing error envelope; no partial fabricated stability. |
| Conflicting signals | Apply priority order; review-sensitive states win. |
| Data source timeout | Existing API error handling applies; no fallback to mock values. |
| Mock mode | Mock fixtures remain explicit and contract-preserving. |

Failure must never expose unauthorized evidence, bypass portfolio membership, or introduce a quality grade.

## 10. Adapter Impact

### Interfaces That Remain Stable

`PassportStore` remains unchanged:

```ts
getPortfolioIndex(sponsorOrgId: string): Promise<PassportPortfolioIndexResponse>
getInstitutionalPassport(sponsorOrgId: string, institutionId: string): Promise<InstitutionalPassport | undefined>
getClaimProvenanceDetail(sponsorOrgId: string, institutionId: string, claimId: string): Promise<PassportClaimProvenanceDetail | undefined>
isInstitutionInPortfolio(sponsorOrgId: string, institutionId: string): Promise<boolean>
```

### Internal Adapter Changes

| Current | Target |
|---|---|
| `DEFAULT_STABILITY = 'Under Review'` in portfolio index | Derived `StabilityIndicator` from Stability Runtime. |
| `STUB_STABILITY = 'Under Review'` in passport detail | Derived `StabilityIndicator` from the same runtime. |
| Stability logic absent from adapter | Dedicated stability module or mapper consumed by `EvidenceCorePassportStore`. |

### Contract Preservation

The public field remains:

```ts
stability: StabilityIndicator
```

No optional explanation field is exposed in RC-12.2 unless separately approved through contract governance.

---

## 11. Runtime Impact

### Portfolio Index

Portfolio index summaries should use the same stability rules as passport detail where feasible.

If the index does not yet load every input needed for full detail stability, implementation has two options:

1. load sufficient Evidence Core data for deterministic stability; or
2. explicitly document and test a conservative index-level stability derivation that never marks uncertain state as `Stable`.

### Passport Detail

Passport detail should use the full available institution read:

- claims;
- evidence nodes by claim;
- organization/audit records where needed;
- mapped passport claims and capabilities.

### Recommendations

Recommendations remain separate. Stability may share inputs with recommendations, but it must not become a next-action engine.

## 13. Exit Criteria

RC-12.2 is complete only when:

- `Under Review` is no longer a universal Evidence Core-mode stub;
- each stability value is derived from deterministic knowledge-state rules;
- the allowed enum remains unchanged;
- no new public fields are exposed;
- Sponsor UI behavior remains contract-compatible;
- stability is not a score, rank, certification, badge, or recommendation;
- contested/review states remain visible;
- stale/decayed evidence does not silently appear current;
- adapter and store boundaries remain intact;
- all technical gates pass;
- completion report records decisions, tests, inherited findings, and residual risks.

## 15. Architectural Impact Assessment

| Question | Expected Answer |
|---|---|
| Does this change RC-10.2? | No. Uses existing `StabilityIndicator` enum. |
| Does this change KUX-008? | No. Implements existing knowledge-state semantics. |
| Does this change Sponsor Platform Architecture? | No. Stays behind `PassportStore` and adapter boundaries. |
| Does this change RC-12.1 Repository Layer? | No. Portfolio remains access scope only. |
| Does this affect Evidence Core policy? | No. Reads existing facts; does not mutate claims/evidence. |
| Does this introduce scoring? | No. Explicitly forbidden. |
| Does this require an ADR? | Not if this specification is followed; yes if enum meaning, public fields, or dependency rules change. |

---

## 16. Expected Result

RC-12.2 should produce this transition:

```
Universal Under Review placeholder
        |
        v
Deterministic Stability Runtime
        |
        v
Existing StabilityIndicator enum
        |
        v
Sponsor Passport Contract unchanged
```

From the user's perspective, the field remains the same. Internally, the value becomes evidence-backed, lifecycle-aware, deterministic, and explainable through existing Sponsor Passport data.
